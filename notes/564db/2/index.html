NORMAL FORMS

ARMSTRONG’S AXIOMS

  • Reflexivity rule: X → a subset of X
  • Augmentation rule: X → Y, then XZ → YZ
  • Transitivity rule: X → Y and Y → Z then X→Z
  • Union rule: X→Y and X→Z, then X→YZ (X, Y, Z are sets of attributes)
  • Decomposition rule: X → YZ, then X → Y and X → Z
  • Pseudo-transitivity rule: X → Y and YZ → U, then XZ → U

DEFINITIONS

Closure: $S^+$ = all FDs logically implied by S

Simple attribute: An attribute that cannot be further subdivided into components is a simple attribute.

Composite attribute: An attribute that can be split into components is a composite attribute.

Non-prime attribute: An attribute that is not part of any candidate key.

Prime attribute: An attribute that is a part of one of the candidate keys.

Trivial functional dependency: If X → Y and Y is the subset of X,

e.g. AB → B, B is subset of AB; A → A, A is subset of A

Non-Trivial functional dependency: If X → Y and Y is not a subset of X,

e.g. AB → C, C is not subset of AB

Partial functional dependency: the non-prime attribute is functionally dependent on part of a candidate key.

e.g. AC → B, A → D, D → B; [A]+ = {A, B, D} then B is partially dependent on AC

Transitive Functional Dependency: dependent is indirectly dependent on determinant

e.g. If A → B and B → C, then A → C is Transitive FD

Multivalued Functional Dependency: entities of the dependent set are not dependent on each other.

e.g. If A → BC and there exists no FD between B and C

Candidate Key & Super Key:

Candidate Key:

  • functionally determines all attributes of R
  • none of its subsets determines all attributes of R

Super Key:

  • set of attributes that contains a candidate key

Properties:

  • A Candidate Key is a minimum SuperKey

  • A Candidate Key is a SuperKey

  • Steps to find Key Attributes:

    R(A, B, C, D, E), FD = {A → B, BC → E, DE → A}

    1. Browse attributes on the right hand of each FD, {B, E, A} are already functionally depended.
    2. Then rest attributes {C, D} are key attributes.
    3. With Key attributes, we can check each combination to find candidate keys.
      1. [CD]+ = {C, D} → not key
      2. [ACD]+ = {A, B, C, D, E} → candidate key
      3. [BCD]+ = {B, C, D, E, A} → candidate key
      4. [CDE]+ = {C, D, E, A, B} → candidate key

  • Example:

    R(A, B, C, D, E), FD = {A → B, BC → E, DE → A}

    C, D will be key attributes since they are not depended on any attributes.

    [CD]+ = {C, D} → not key

    [ACD]+ = [BCD]+ = [CDE]+ = {A, B, C, D, E} → candidate keys are found

    Candidate keys: ACD, BCD, CDE

    Super Keys (must include CD, then find all combinations):

    ACD, BCD, CDE, ABCD, BCDE, ACDE, ABCDE

NORMAL FORMS

First Normal Form (1NF):

  • Every field must contain atomic values (i.e no sets or lists).

  • No null values.

  • Has unique key.

  • All relations are in 1NF.

  • Example:

    Not Atomic: R(Name, Course): (steve, c/c++)

    Atomic: R(Name, Course): (steve, c), (steve, c++)

Second Normal Form (2NF):

  • The table should be in the First Normal Form (1NF).
  • There should be No Partial Dependency for non-prime attributes.

Lemmas:

  • R is in 2NF if it does not contain any composite key (implies no partial dependency) and is in 1NF

  • Example:

    R(A, B, C, D), FD = {AC → B, A → D, D → B}

    We can compute the closure of [A]+ = {A, D, B}

    Therefore, A is alone capable of determining B, which means B is Partially Depends on AC.

    To decompose R and FD, we could take B out of R

    R1(A, B, C), FD = {AC → B}

    R2(A, B, D), FD = {A → D, D → B)

    Then there is no violation in both R1 and R2.

Third Normal Form (3NF):

  • The table should be in the Second Normal Form (2NF).
  • No Transitive Dependency for non-prime attributes

Lemmas (for quick identification): at least one of the following must be true

  1. X → Y is a Trivial FD
  2. X is SuperKey for R
  3. Y is part of a candidate key

  • Advantages:

    • 3NF is considered adequate for normal relational database design
    • most of the 3NF tables are free of insertion, update, and deletion anomalies.
    • 3NF always ensures functional dependency preserving and lossless

  • Example:

    R(A, B, C, D, E), FD = {A → B, A → C, C → D, A → E}

    Since A is missing on the right side of FDs, A is a key attribute.

    [A]+ = {A, B, C, D, E}, then A is candidate key.

    However, A → C & C → D implies A → D, then D is transitively dependent on A

    To decompose to R to 3NF, Take D out of R,

    R1(C, D), FD = {C → D}

    R2(A, B, C, E), FD = {A → B, A → C, A → E}

    This decomposition also in BCNF

Boyce-Codd Normal Form (BCNF) or (3.5NF):

  • The table should be in the Third Normal Form (3NF).
  • For every one of its dependencies X → Y, at least one of the following conditions hold
    1. X → Y is a trivial FD
    2. X is superkey of R

Other Lemmas:

  • Any 2-attribute relation is in BCNF.

  • BCNF decomposition may always not possible with dependency preserving, however, it always satisfies lossless join condition

  • Example:

    R(A, B, C, D, E), FD = {AB → CD, D → E, A → C, B → D}

    [AB]+ = {A, B, C, D, E}, then AB is candidate key

    D → E Violation: Transitive Dependency; D is not a Super key

    A → C Violation: Partial Dependency; A is not a Super key

    B → D Violation: Partial Dependency; B is not a Super key

    1. resolve D → E:

      R1(D, E), FD = {D → E}, (D is superkey)

      R2(A, B, C, D), FD = {AB → CD, A → C, B → D}

    2. resolve A → C and B → D: break R2

      R3(A, C), FD = {A → C}, (A is superkey)

      R4(B, D), FD = {B → D}, (B is superkey)

    Final Decomposition: R1(D, E), FD = {D → E}, R3(A, C), FD = {A → C}, R4(B, D), FD = {B → D}

    In above decomposition AB → CD is missing which is in the original relation. however we can derive AB → CD as below:

    AB → AB ⇒ AB → A and AB → B

    AB → A and A → C ⇒ AB → C

    AB → B and B → D ⇒ AB → D

    AB → C and AB → D ⇒ AB → CD

    Hence, dependency also preserved.

Other NFs:

Fourth Normal Form (4NF):

  • The table should be in the Boyce-Codd Normal Form (BCNF)
  • No Multi-valued Dependency.

Fifth Normal Form (5NF):

  • The table should be in the Fourth Normal Form (4NF).
  • Does not contain any join dependency, joining should be lossless.

Desirable Properties of Schema Decomposition

  1. MINIMIZE REDUNDANCY
  2. AVOID INFO LOSS
  3. PRESERVE DEPENDENCY
  4. ENSURE GOOD QUERY PERFORMANCE

For R (Relation) with S (FDs)→ decomposition → R1 with S1, R2 with S2

  • LOSSLESS-JOIN DECOMPOSITION

    • Lossless join decomposition is a decomposition of a relation R into relations R1, R2 such that if we perform a natural join of relation R1 and R2, it will return the original relation R.
    • S+ contains one of FDs: R1 ∩ R2 → R1 or R1 ∩ R2 → R2

  • DEPENDENCY PRESERVING

    • S+ = (S1 U S2)+
    • general form: S+ = S’+, S’ = (S1 U S2 U … U Sn)

  • Example:

    R(A, B, C) S = {A → B, B → C}

    R is not in BCNF since B is not superkey

    Decompose: R1(B, C), R2(A, B)

    • Lossless-join decomposition:

      R1 ∩ R2 = {B} and B+ = {B, C}, then B → BC

    • Dependency preserving:

      A → B is enforced in R2 B → C is enforced in R1

    Therefore, this decomposition is loss-less join and dependency preserving

BCNF and 3NF

  • BCNF removes certain types of redundancy
  • For examples of redundancy that it cannot remove, see “multi-valued redundancy”
  • BCNF avoids info loss
  • BCNF is not always dependency preserving
  • In fact, some times we cannot find a BCNF decomposition that is dependency preserving

When?

  • BCNF is not dependency preserving
  • Efficient checking for FD violation on updates is important

Define a weaker normal form, called Third Normal Form (3NF)

  • Allows some redundancy
  • But functional dependencies can be checked on individual relations without computing a join.
  • There is always a lossless-join, dependency-preserving decomposition into 3NF.

BCNF vs 3NF

It is always possible to decompose a relation into a set of relations that are in

  • 3NF
    • the decomposition is lossless
    • the dependencies are preserved
  • BCNF
    • the decomposition is lossless
    • it may not be possible to preserve dependencies.

DECOMPOSING R INTO 3NF

  1. Get a “minimal cover” of FDs

    OBTAINING MINIMAL COVER

    1. Put the FDs in a standard form (i.e. right-hand side should contain only single attribute)
    2. Minimize the left side of each FD
    3. Delete redundant FDs

    • Example

  2. Find a lossless-join decomposition of R (which might miss dependencies)

  3. Add additional relations to the decomposition to cover any missing FDs of the cover

Result will be lossless, will be dependency-preserving 3NF; might not be BCNF

NORMALIZATION SUMMARY

  • 1NF: usually part of the woodwork
  • 2NF: usually skipped
  • 3NF: a biggie­ always aim for this
  • BCNF and 4NF: tradeoffs start here ­ in re: d-preserving and losslessness
  • 5NF: You can say you’ve heard of it…
