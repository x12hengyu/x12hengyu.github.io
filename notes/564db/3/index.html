Data Definition Language (DDL):

  • Define relational schemas
  • Create/alter/delete tables and their attributes
// Create table student
// Student(snum:int, sname:String, level:String, major:String, age:integer)
CREATE TABLE Student (
	snum BIGINT,
	sname VARCHAR(50), 
	major VARCHAR(50), 
	level CHAR(2), 
	age int,
	PRIMARY KEY (snum));

// Delete table enroll
DROP TABLE enroll;

// Add a foreign key constraint to table enroll
ALTER TABLE enroll
ADD FOREIGN KEY(snum) 
REFERENCES STUDENT(snum) 
ON UPDATE CASCADE;

Standard SQL requires these distinctions:

  • There can be only one PRIMARY KEY for a relation, but several UNIQUE attributes.
  • No attribute of a PRIMARY KEY can ever be NULL in any tuple. But attributes declared UNIQUE may have NULL’s, and there may be several tuples with NULL**.**

Two other declarations we can make for an attribute are:

  • NOT NULL means that the value for this attribute may never be NULL.
  • DEFAULT says that if there is no specific value known for this attribute’s component in some tuple, use the stated .

Constraints:

  • DEFAULT - Sets a default value for a column if no value is specified

    • e.g.

      CREATE TABLE CLASS (
	cname VARCHAR(255), 
	meets_at VARCHAR(255),
	room VARCHAR(255) DEFAULT 'R128’,
	fid bigint,
	PRIMARY KEY(cname));

  • NOT NULL - Ensures that a column cannot have a NULL value

  • UNIQUE - Ensures that all values in a column are different

  • PRIMARY KEY - A combination of a NOT NULL and UNIQUE. Uniquely identifies each row in a table

  • FOREIGN KEY - Prevents actions that would destroy links between tables

    • If there is a foreign-key constraint from attributes of relation R to the primary key of relation S, two violations are possible:

      1. An insert or update to R introduces values not found in S.
      2. A deletion or update to S causes some tuples of R to “dangle.”

    • Suppose R and S, to handle nonexistent S must be rejected:

      1. Default: Reject the modification.

      2. Cascade: Make the same changes in R.

        Deleted Class: delete R tuple.

        Updated Class: change value in R.

      3. Set NULL: Change S to NULL.

    • When we declare a foreign key, we may choose policies SET NULL or CASCADE independently for deletions and updates: ON [UPDATE, DELETE][SET NULL CASCADE]

    • e.g.

      // Create table ENROLL
// Enroll (snum:integer, cname:String)
CREATE TABLE ENROLL ( 
	snum BIGINT,
	cname VARCHAR(255),
	FOREIGN KEY (snum) REFERENCES student(snum),
  FOREIGN KEY (cname) REFERENCES class(cname) on UPDATE CASCADE,
	primary key(snum, cname));

// FOREIGN KEY on ALTER TABLE
ALTER TABLE Orders
ADD FOREIGN KEY (PersonID) REFERENCES Persons(PersonID);

  • CHECK - Ensures that the values in a column satisfies a specific condition

  • CREATE INDEX - Used to create and retrieve data from the database very quickly

NULL VALUE

  • The logic of conditions in SQL is really 3-valued logic: TRUE, FALSE, UNKNOWN.
  • When any value is compared with NULL, the truth value is UNKNOWN.
  • But a query only produces a tuple in the answer if its truth value for the WHERE clause is TRUE (not FALSE or UNKNOWN).
  • To understand how AND, OR, and NOT work in 3-valued logic, think of TRUE = 1, FALSE = 0, and UNKNOWN = 1⁄2.
  • X AND Y = MIN(X, Y); X OR Y = MAX(X, Y), NOT(X) = 1-X.
  • NULL never contributes to a sum, average, or count, and can never be the minimum or maximum of a column.
  • But if there are no non-NULL values in a column, then the result of the aggregation is NULL.

We may follow a SELECT-FROM-WHERE expression by GROUP BY and a list of attributes.

RELATIONAL ALGEBRA

  • Union: U

    • Input: R1, R2, must have the same schema
    • Output: R1 U R2 has the same schema as R1, R2
    • Union: all tuples in R1 or R2
    • Notation: R1 U R2

  • Difference: -

    • Input: R1, R2, must have the same schema
    • Output: R1 - R2 has the same schema as R1, R2
    • Difference: all tuples in R1 and not in R2
    • Notation: R1 – R2

  • ­Selection: $\sigma$

    • Input: A relation R, a condition c

    • Output: a relation $R^\prime = \sigma_c(R)$ which contains all tuples from R satisfying a condition c and R’ has the same schema as R

    • c is a condition: =, <, >, and, or, not

    • e.g.

      Find all employees with salary more than $40,000: ­$\sigma_{salary\ >\ 40000}\ (Employee)$

  • Projection: π­

    • Input: a relation R(B1, …, Bm) and a set of attributes A1, A2, …, An.

    • Output: A relation R’(A1,A2,..An) in which: ${A1, …, An}\ \subseteq\ {B1, …, Bm}$

    • Unary operation: returns certain columns A1, A2, …, An

    • Eliminates duplicate tuples !

    • Notation: $\pi_{A1,…,An}(R)$

    • e.g.

      project social-security number and names: ­$\pi_{ssn,\ name}\ (Employee)$

  • Cartesian Product: X

    • Input: $R1(A1,…,An),\ R2(B1,…,Bm), \ {{A1,…,An}} \cap {{B1,…Bm}} = \phi$

    • Output: Return the concatenation of every tuple in R1 with every tuple in R2

    • Notation: R1 x R2

    • Output schema is S(A1, …, An, B1, …, Bm)

    • e.g.

      Employee x Dependents

  • Renaming**: $\rho$**

    • Input: Input schema: R(A1, …, An)

    • Output: Output schema: S(B1, …, Bn)

      • Does not change the relational instance
      • Changes the relational schema only

    • Notation: $\rho_{B1,…,Bn}(R)$

    • e.g.

      $\rho_{LastName,\ SocSocNo}(Employee)$

  • Intersection­: $\cap$

    • Input: R1, R2 must have the same schema
    • Output:
      • $R1\ \cap\ R2$ has the same schema as R1, R2 ­
      • All tuples both in R1 and in R2
    • Notation: $R1\ \cap\ R2$
    • Intersection is derived: $R1\ \cap\ R2= R1-(R1-R2)$

JOINS (THETA-JOIN, NATURAL-JOIN, EQUI-JOIN, OTHER JOINS)

  • THETA-JOIN:

    • Notation: $R1 \bowtie_\theta R2$ where $\theta$ is the condition

    • Input schemas: R1(A1,…,An), R2(B1,…,Bm)

    • Output schema: S(A1,…,An,B1,…,Bm)

    • Derived operator: $R1 \bowtie_\theta R2 = \sigma_\theta(R1\times R2)$

    • e.g.

  • NATURAL-JOIN:

    • Notation: $R1 \bowtie R2$

    • Input Schema: R1(A1, …, An), R2(B1, …, Bm)

    • Output Schema: *S(C1,…,Cp)*­ Where {C1, …, Cp} = {A1, …, An} U {B1, …, Bm}

    • Meaning: combine all pairs of tuples in R1 and R2 that agree on the attributes: ${{A1,…,An}} \cap {{B1,…Bm}}$ (called the join attributes)

    • e.g.

      ****Given the relations R(A, B, C, D) and S(B, D, E):

      $r \bowtie s = π_{r.A,r.B,r.C,r.D,s.E}(σ_{r.B=s.B∧r.D=s.D}(r × s))$

  • EQUI-JOIN:

    • Return all the combinations of tuples of R1 and R2 which satisfy the equality condition A=B

      • ­Equi-join is a special case of theta join
      • Natural join is a special case of equi-join

    • Most frequently used in practice: $R1 \bowtie_{A=B} R2$
